# 必须遵守的开发约束

## 1. 系统定位
这是一个面向医院临床场景的临床辅助决策系统。
系统包含：
- 医生PC端（独立知识查询端）
- HIS嵌入端（诊疗辅助端）
- 移动端（App）
- 管理端
- 平台服务层

其中：
- 医生PC端与HIS嵌入端不是两个割裂系统，而是同一产品的两个入口
- HIS嵌入端允许有轻量搜索、疾病查看、药品查看、指南查看
- 医生PC端承载更完整的知识门户能力
- 两者共用同一套知识库与服务层

## 2. 开发架构要求
请严格按照以下分层架构设计：
- DB层
- Services层
- Tools/API层
- GUI层

要求你在架构文档中明确说明每一层的职责、目录结构、调用关系、边界约束。

## 3. 技术方向
- 后端使用 Python
- 优先采用适合快速开发和后续扩展的 Python Web 框架
- 前端/GUI层请根据实际情况选择合适方案，但要说明原因
- 如果某些端暂时不适合一次性全部开发，请先定义统一架构，再分阶段落地

## 5. 开发原则
- 不要把所有招标功能一次性做完
- 先完成最小可演示闭环（MVP）
- 优先建设 HIS嵌入端 + 医生PC端基础知识检索 + 管理端最小后台 + 平台服务层基础能力
- 移动端可以后置
- 先用 Mock 数据跑通主流程，再预留真实 HIS/LIS/PACS/EMR 对接接口
- 功能说明尽量贴近实际开发，而不是纯投标话术
