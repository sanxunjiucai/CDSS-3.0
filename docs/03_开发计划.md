# 临床辅助决策系统（CDSS 3.0）开发计划

**版本：** v1.0
**日期：** 2026-03-12
**状态：** 已确认

---

## 总体原则

- 以 **HIS嵌入端全量 + PC端基础 + 管理端最小 + 服务层基础** 为 MVP 目标
- 外部系统（HIS/LIS/PACS/EMR）全程使用 Mock，第三阶段后期才切换真实对接
- 每阶段结束必须有可演示的闭环功能，不做半成品
- 前后端并行开发，以 API 契约（OpenAPI Schema）作为协作界面

---

## 第一阶段：项目骨架 + 基础知识服务

**目标：** 搭建完整工程骨架，跑通最基础的知识查询闭环（搜索 → 详情）

### 后端任务

| 任务 | 说明 |
|-----|------|
| 工程初始化 | FastAPI 项目骨架、目录结构、Docker Compose（PG + ES + Redis） |
| 数据库初始化 | SQLAlchemy 模型定义（Disease、Drug、Exam、User），Alembic 初始化迁移 |
| Elasticsearch 索引 | 定义疾病/药品/检验索引，IK中文分词配置 |
| 基础 CRUD API | 疾病/药品/检验 的列表、详情接口（GET） |
| 全局检索 API | `GET /v1/search?q=xxx`，跨库联合搜索 |
| JWT 认证 | 登录接口、Token 验证中间件 |
| 统一响应格式 | 全局异常处理、统一 Response 结构 |
| Mock 数据 | 50条疾病、50条药品、50条检验的 Seed 数据 |

### 前端任务

| 任务 | 说明 |
|-----|------|
| 三端工程初始化 | Vue3 + Vite 初始化（pc / his / admin），共享 shared 目录 |
| 共享 API 封装 | axios 封装，统一 Token 注入、错误处理 |
| PC端基础页面 | 首页工作台、全局搜索页、分类导航、疾病/药品/检验详情页 |
| 管理端登录 | 登录页 + JWT 存储 |
| 管理端知识库列表 | 疾病/药品/检验 CRUD 页面（列表 + 新增 + 编辑 + 删除） |

### 阶段产出
- 可运行的后端服务（Docker Compose 一键启动）
- PC端可搜索疾病/药品/检验，查看详情
- 管理端可登录并进行知识库基础管理
- Postman / OpenAPI 文档可用

### 依赖关系
- 无外部依赖，全程 Mock 数据

## 第二阶段：HIS嵌入端全量功能

**目标：** 完成 HIS嵌入端全部 23 个功能模块，所有外部数据使用 Mock

| 任务 | 说明 |

| Mock HIS 适配器 | 返回模拟患者信息（基本信息、诊断、过敏史） |
| Mock LIS 适配器 | 返回模拟检验结果（含异常值） |

| 治疗方案服务 | 诊断 → 标准治疗路径（知识库关联） |
| 个性化治疗服务 | 结合患者画像（年龄/过敏/肾功能）过滤调整推荐结果 |
| 检验推荐服务 | 诊断 → 推荐检验组合（知识库关联规则） |
| 检验结果解读服务 | 比对参考范围，生成异常解读文本 |
| 多参考值判断 | 根据患者性别/年龄选择对应参考范围段 |
| 风险评估/量表服务 | 量表 CRUD + 自动评分计算接口 |
| 诊断合理性审核 | 诊断 + 处置 → 规则校验（药物禁忌、适应症不符等） |
| 实时预警规则 | 预警规则配置 + 触发检测接口 |
| 转诊评估服务 | 基于诊断/风险评分给出转诊建议文本 |
| 循证医学参考 | 诊断 → 关联指南/文献列表 |
| 健康干预建议 | 疾病 → 生活建议文本（知识库字段） |
| 指南文献服务 | 指南列表、详情接口 |
| 开单辅助接口 | 检验/检查开单时查询适应症和准备事项 |
| 报告辅助接口 | 报告查看时侧边调用知识解读 |

## Mock 与真实接口规划

### 必须使用 Mock 的模块（第一~三阶段）
- HIS 患者上下文（患者信息、当前诊断）
- LIS 检验结果
- PACS 影像报告
- EMR 病历数据
- 心电/病理报告

### 必须预留真实接口的模块（即使MVP也要设计好接口）
- `BaseHISAdapter` 接口定义（第一阶段完成）
- `BaseLISAdapter` 接口定义（第一阶段完成）
- 所有适配器通过工厂函数注入，Mock/真实由 `.env` 切换

### 可以直接实现不需要Mock的模块
- 知识库 CRUD（内部数据）
- 量表评分计算（规则引擎，无外部依赖）
- 全文检索（Elasticsearch 内部）
- JWT 认证与权限控制

## 开发约定

### 接口开发规范
1. 先定义 Pydantic Schema（请求/响应模型），再写 Service，最后写 Router
2. 所有接口必须有 OpenAPI 描述（summary、description、response_model）
3. Mock 适配器与真实适配器实现相同接口，Service 层不区分

### 前后端协作规范
1. 后端先出接口文档（FastAPI 自动生成），前端按文档对接
2. 未完成的后端接口，前端临时使用本地 mock.json 文件占位
3. 所有接口变更同步更新文档

### 代码质量规范
1. 后端：black 格式化，flake8 检查，每个 Service 方法写类型注解
2. 前端：ESLint + Prettier，组件使用 Composition API
3. 提交前必须本地通过单元测试
