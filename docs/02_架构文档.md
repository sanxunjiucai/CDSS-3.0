## 一、总体架构说明

CDSS 3.0 采用**前后端分离、分层服务**架构，整体分为四层：

```
┌─────────────────────────────────────────────────────────┐
│                      GUI 层                              │
│   医生PC端(Web)  HIS嵌入端(iframe)  管理端(Web)  App(后置) │
└─────────────────────┬───────────────────────────────────┘
                      │ HTTP / REST
┌─────────────────────▼───────────────────────────────────┐
│                  Tools / API 层                          │
│         统一 REST API 网关（FastAPI）                     │
│   认证鉴权  路由分发  参数校验  限流  日志  Mock切换开关    │
└─────────────────────┬───────────────────────────────────┘
                      │ 函数调用
┌─────────────────────▼───────────────────────────────────┐
│                  Services 层                             │
│  疾病服务  药品服务  检验服务  指南服务  辅助诊断服务        │
│  风险评估服务  检验解读服务  诊断审核服务  外部对接适配器    │
└──────────────┬────────────────┬────────────────────────┘
               │                │
┌──────────────▼──┐      ┌──────▼──────────────────────┐
│    DB 层         │      │      外部系统（Mock/真实）      │
│  PostgreSQL      │      │  HIS  LIS  PACS  EMR  心电  │
│  Elasticsearch  │      └─────────────────────────────┘
│  Redis           │
└─────────────────┘
```

## 二、技术栈

### 后端
| 技术 | 选型 | 理由 |
|-----|------|------|
| 语言 | Python 3.11+ | 指定要求，生态丰富 |
| Web框架 | FastAPI | 异步高性能，自动生成 OpenAPI 文档，类型安全，适合 API 开发 |
| ORM | SQLAlchemy 2.0 + Alembic | 成熟稳定，支持异步，Alembic 管理数据库迁移 |
| 全文检索 | Elasticsearch 8.x | 医疗知识库全文检索，支持中文分词（IK分词器） |
| 缓存 | Redis 7.x | 热点知识缓存、会话存储、速率限制 |
| 关系数据库 | PostgreSQL 15 | 主数据库，结构化数据存储 |
| 任务队列 | Celery + Redis | 知识库异步导入、更新任务 |
| 认证 | JWT (python-jose) | 无状态认证，适合多端 |

### 前端
| 端 | 技术 | 理由 |
|---|------|------|
| 医生PC端 | Vue 3 + Vite + Element Plus | 适合知识门户类产品，组件库成熟，开发效率高 |
| HIS嵌入端 | Vue 3 + Vite（轻量打包） | 与PC端同框架共享组件，iframe嵌入无冲突 |
| 管理端 | Vue 3 + Vite + Element Plus | 同框架，共享管理组件 |
| 移动端 | 后置，预留（React Native 或 uni-app） | — |

### 基础设施
| 技术 | 用途 |
|-----|------|
| Docker + Docker Compose | 本地开发与部署 |
| Nginx | 前端静态托管 + API 反向代理 |
| MinIO | 知识库文件存储（指南PDF、图片等） |

## 三、分层架构设计

### 3.1 DB 层

**职责：** 数据持久化与检索，不包含任何业务逻辑。

**组成：**
- **PostgreSQL**：存储结构化数据，包括知识库实体（疾病、药品、检验等）、用户、配置、日志
- **Elasticsearch**：全文检索索引，同步自 PostgreSQL，支持中文分词
- **Redis**：缓存层（热点详情页）、会话存储、Celery Broker

**边界约束：**
- Services 层通过 Repository 模式访问 DB，不允许 API 层直接操作 DB
- Elasticsearch 仅用于检索，不作为数据主存储，主数据以 PostgreSQL 为准
- DB 层不感知业务规则

**目录：**
```
backend/
└── db/
    ├── models/         # SQLAlchemy 数据模型
    ├── repositories/   # Repository 封装（每个实体一个文件）
    ├── migrations/     # Alembic 迁移文件
    └── elasticsearch/  # ES 索引定义与同步逻辑
```

---

### 3.2 Services 层

**职责：** 业务逻辑的唯一承载层，处理数据组装、规则计算、外部适配器调用。

**主要服务模块：**

| 服务 | 文件 | 职责 |
|-----|------|------|
| DiseaseService | disease_service.py | 疾病查询、详情、分类 |
| DrugService | drug_service.py | 药品详情、禁忌、相互作用 |
| ExamService | exam_service.py | 检验检查说明、参考范围 |
| GuidelineService | guideline_service.py | 指南列表、全文 |
| SearchService | search_service.py | 跨库统一检索 |
| DiagnosisService | diagnosis_service.py | 辅助诊断（症状→诊断推荐） |
| TreatmentService | treatment_service.py | 治疗方案推荐 |
| LabResultService | lab_result_service.py | 检验结果解读 |
| RiskAssessService | risk_assess_service.py | 量表评分、风险评估 |
| AuditService | audit_service.py | 诊断合理性审核、预警规则 |
| PatientContextService | patient_context_service.py | 患者上下文接入与缓存 |
| AdapterService | adapters/ | 外部系统适配器（HIS/LIS/PACS/EMR等） |

**边界约束：**
- Services 层只调用 Repositories 和 Adapters，不直接处理 HTTP Request/Response
- 业务规则只在 Services 层实现，不允许在 API 层写业务判断
- 外部系统调用统一通过 AdapterService，Mock/真实模式由配置切换

**目录：**
```
backend/
└── services/
    ├── disease_service.py
    ├── drug_service.py
    ├── exam_service.py
    ├── guideline_service.py
    ├── search_service.py
    ├── diagnosis_service.py
    ├── treatment_service.py
    ├── lab_result_service.py
    ├── risk_assess_service.py
    ├── audit_service.py
    ├── patient_context_service.py
    └── adapters/
        ├── base_adapter.py     # 适配器基类（定义接口）
        ├── his_adapter.py      # HIS 适配器（Mock/真实）
        ├── lis_adapter.py      # LIS 适配器
        ├── pacs_adapter.py     # PACS 适配器
        ├── emr_adapter.py      # EMR 适配器
        ├── ecg_adapter.py      # 心电适配器
        └── pathology_adapter.py # 病理适配器
```

### 3.3 Tools / API 层

**职责：** 对外暴露 REST API，处理 HTTP 协议相关事务（认证、路由、参数校验、响应格式化）。

**实现：FastAPI Router 模块化组织**

**主要 Router 模块：**
```
backend/
└── api/
    ├── v1/
    │   ├── auth.py             # 登录、Token刷新
    │   ├── search.py           # 全局检索
    │   ├── disease.py          # 疾病相关接口
    │   ├── drug.py             # 药品相关接口
    │   ├── exam.py             # 检验检查接口
    │   ├── guideline.py        # 指南文献接口
    │   ├── diagnosis.py        # 辅助诊断接口
    │   ├── treatment.py        # 治疗方案接口
    │   ├── lab_result.py       # 检验结果解读接口
    │   ├── risk_assess.py      # 风险评估/量表接口
    │   ├── audit.py            # 诊断审核/预警接口
    │   ├── patient.py          # 患者上下文接口
    │   └── admin/              # 管理端专用接口
    │       ├── knowledge.py    # 知识库管理
    │       ├── config.py       # 配置管理
    │       └── account.py      # 账号权限管理
    ├── deps.py                 # 依赖注入（当前用户、DB Session）
    └── middleware.py           # 日志、限流、异常处理中间件
```

**边界约束：**
- API 层只调用 Services 层，禁止直接操作 DB
- 所有响应使用统一格式：`{ code, message, data }`
- 接口版本化（v1），为后续升级预留

---

### 3.4 GUI 层

**职责：** 用户界面，消费 API 层接口，不包含业务逻辑。

**组成：**
```
frontend/
├── pc/         # 医生PC端（Vue3）
├── his/        # HIS嵌入端（Vue3，独立打包）
└── admin/      # 管理端（Vue3）
```

**边界约束：**
- 前端只通过 API 获取数据，不直接访问 DB 或 Services
- HIS嵌入端通过 iframe 嵌入，接收 HIS postMessage 传入患者上下文
- 三端共享 `shared/` 组件库（知识卡片、搜索框等公共组件）

## 六、数据流转关系

### 检索流程
```
前端搜索请求
→ API层 /v1/search (GET ?q=xxx)
→ SearchService.search(keyword)
→ ES全文检索（疾病/药品/检验索引）
→ 返回聚合结果列表
→ 前端渲染分类结果
```

### HIS嵌入端患者上下文流程
```
HIS页面 postMessage(patient_info)
→ HIS嵌入端前端接收
→ API层 POST /v1/patient/context
→ PatientContextService.set_context(patient_info)
→ Mock/真实HIS适配器补全数据
→ Redis缓存（TTL 30min）
→ 后续接口携带 patient_id 获取上下文
```

### 检验结果解读流程
```
前端提交检验结果（Mock LIS数据）
→ API层 POST /v1/lab-result/interpret
→ LabResultService.interpret(results, patient_context)
→ 查询 Exam 参考范围 → 比对异常值
→ 调用知识库获取临床意义文本
→ 返回解读报告（正常/偏高/偏低 + 临床意义 + 建议）
```

## 八、外部系统对接设计

### 适配器模式
所有外部系统通过统一的适配器接口访问，Services 层不感知 Mock 还是真实：

```python
# base_adapter.py
class BaseHISAdapter:
    async def get_patient_info(self, patient_id: str) -> PatientContext:
        raise NotImplementedError

# his_adapter.py
class MockHISAdapter(BaseHISAdapter):
    async def get_patient_info(self, patient_id: str) -> PatientContext:
        return MOCK_PATIENT_DATA[patient_id]

class RealHISAdapter(BaseHISAdapter):
    async def get_patient_info(self, patient_id: str) -> PatientContext:
        # 调用真实 HIS WebService/REST 接口
        ...
```

### Mock / 真实切换方案
通过环境变量控制，无需修改代码：

```env
# .env
HIS_MODE=mock          # mock | real
LIS_MODE=mock
PACS_MODE=mock
EMR_MODE=mock
```

```python
# config.py 工厂函数
def get_his_adapter() -> BaseHISAdapter:
    if settings.HIS_MODE == "real":
        return RealHISAdapter(settings.HIS_API_URL)
    return MockHISAdapter()
```

### 外部系统接口预留规范
每个适配器定义标准接口方法，真实对接时只需实现对应 Adapter 类：

| 适配器 | 核心接口方法 |
|-------|------------|
| HIS | get_patient_info, get_current_diagnosis, get_prescriptions |
| LIS | get_lab_results, get_result_by_id |
| PACS | get_report, get_imaging_report |
| EMR | get_medical_history, get_chief_complaint |
| 心电 | get_ecg_report |
| 病理 | get_pathology_report |

## 十、关键技术决策说明

| 决策点 | 选择 | 理由 |
|-------|------|------|
| 为何用 FastAPI 而非 Django | FastAPI | 异步原生支持，自动 API 文档，类型提示驱动，适合纯 API 服务 |
| 为何用 Elasticsearch | 医疗知识库全文检索复杂（ICD编码、别名、中文分词），PostgreSQL 全文检索能力不足 | — |
| HIS嵌入端为何用 iframe | 避免与 HIS 前端框架冲突，隔离 CSS/JS，部署简单，HIS厂商改造成本低 | — |
